# 周杰伦 Apple Music 桌宠产品需求文档

版本：`v0.1`
日期：`2026-05-04`
状态：`只读调研版`

## 1. 项目一句话

做一个独立的 macOS 桌面产品：当用户在 Apple Music 播放周杰伦歌曲时，桌面上会出现并切换到对应专辑的 Q 版像素人物；用户可以点击、拖动、语音控制它，并通过它完成常用的音乐操作与轻陪伴互动。

这不是一个“给 Codex 换皮肤”的小功能，而是一个以 `Apple Music + 桌宠 + 像素角色 + 轻 Agent` 为核心的独立产品。

## 3. 产品定位

### 3.1 产品本质

它本质上应该是一个：

- `音乐感知桌宠`
- `专辑角色切换器`
- `Apple Music 轻控制器`
- `语音可控的桌面陪伴物`

### 3.2 不应该做成什么

1. 不应该做成完整 Apple Music 替代播放器。
2. 不应该做成纯聊天宠物。
3. 不应该做成只能展示、没有控制能力的动态贴纸。
4. 不应该把产品运行时绑定死在 Codex Pet 上。

### 3.3 最合理的形态

最合理的形态是：

- 前台：一个可拖动、可点击、可切换动作的原生 macOS 桌宠
- 后台：一个持续监听 Apple Music 当前播放状态的小型本地 agent
- 入口：桌宠本体 + menu bar 菜单
- 扩展：后续接语音、歌词、推荐说明、专辑彩蛋

## 4. 这次调研后的关键判断

### 4.1 Codex Pet 能借什么，不能借什么

Codex Pet 目前更像 `Codex 内置悬浮陪伴物`，不是通用桌宠框架。

能借的主要是：

1. 小体积、常驻、低打扰的产品气质
2. 固定状态动画的资源组织方式
3. `pet.json + spritesheet.webp` 这类标准化资产思路

不能把它当成主运行时的原因：

1. 官方公开能力主要是跟随 Codex 自己的状态，不是接外部事件
2. 没看到官方开放 Apple Music 这类外部状态源直连接口
3. 它更像“显示层”，不是“行为层”

结论：

`Codex Pet 适合做视觉参考，不适合直接承载你的主产品逻辑。`

### 4.2 GitHub 同类项目给出的方向

近几年最值得学的，不是老式电子宠物，而是这三条路线的组合：

1. `状态可视化桌宠`
2. `Apple Music companion`
3. `标准化角色动画资产`

最有参考价值的代表项目：

1. `Clawd on Desk`：把 agent 状态翻译成桌面角色状态
2. `SpotMenu`：把 Apple Music 做成轻量伴随控制层
3. `Tuneful`：把音乐控制做成原生 macOS 入口
4. `LyricFever`：把播放状态延展成氛围层
5. `VPet`：证明角色内容体系本身也能成为长期价值

结论：

`你的方向不是“再做一个宠物”，而是“把音乐状态、用户动作、角色表现，合成一个长期可玩的桌面伴侣”。`

### 4.3 语音输入法项目里最值得复用的部分

PulseType 那条线里，已经有一套真实可用的 Apple Music 控制能力，适合借到这个产品里。

最值得复用的是：

1. `osascript` 执行与错误处理
2. Apple Music 打开、播放、暂停、继续、上一首、下一首
3. 按曲库 `persistent ID` 精确播放
4. 播放结果校验，不是只发命令不验证结果
5. 语音指令识别后的轻量音乐路由

不建议整块搬的是：

1. 整套输入法级别的交互协调器
2. 大而全的 agent 基建
3. 与音乐无关的文档、提醒、邮件等能力

结论：

`这次最适合借的是音乐控制内核，不是整个语音输入法产品。`

## 5. 核心产品目标

### 5.1 用户价值

用户打开 Apple Music 听周杰伦时，这个桌宠能做到三件事：

1. 立刻让桌面角色和当前歌曲所属专辑对应起来
2. 让“听歌”变成更有陪伴感、更有情绪表达的桌面体验
3. 让常用音乐操作不需要切回 Apple Music 主窗口

### 5.2 产品目标

1. 做到 `听到哪张专辑，就出现哪张专辑角色`
2. 做到 `角色不仅会显示，还会动、会响应、会控制`
3. 做到 `桌宠的动作表达和音乐状态有明确对应关系`
4. 做到 `未来可以继续扩成更完整的周杰伦角色宇宙`

## 6. 核心功能设计

## 6.1 专辑识别与角色切换

### 目标

根据 Apple Music 当前曲目的元数据，识别专辑字段并映射到对应人物角色。

### 规则

1. 监听当前曲目变化
2. 读取 `artist / album / track / playback state`
3. 先校验艺人是否属于周杰伦
4. 再用专辑名命中角色映射表
5. 命中后切换到对应角色资源
6. 未命中时进入通用待机角色或“未知专辑状态”

### 要求

1. 专辑匹配要支持简繁中文、英文名、别名
2. 同一张专辑的不同版本名要能归并
3. 切换不能闪烁，不能每秒重复重载角色

## 6.2 桌宠基础行为

### 必备行为

1. 常驻桌面
2. 可拖动
3. 可停靠屏幕边缘
4. 可点击
5. 可长按或右键呼出操作菜单
6. 可从 menu bar 打开设置

### 状态建议

1. `idle`：暂停、未播放、刚出现
2. `running`：正在播放
3. `waving`：用户点击问候、切歌成功
4. `waiting`：Apple Music 未打开、无周杰伦曲目
5. `review`：语音识别中、准备执行指令
6. `failed`：控制失败、未命中曲库、权限问题
7. `jumping`：点亮爱心成功、切换专辑成功、彩蛋触发

这里可以借鉴 Codex Pet 的固定状态语义，但行为驱动必须由你自己的 app 控制。

## 6.3 Apple Music 控制

### 首批必须有

1. 播放 / 暂停
2. 下一首 / 上一首
3. 打开 Apple Music
4. 显示当前歌曲名与专辑名
5. 曲目切换后同步切角色

### 第二层增强

1. 点桌宠某个区域切歌
2. 右键菜单里直接选“播放上一张专辑 / 下一张专辑”
3. 从桌宠发起“播放这张专辑代表作”
4. 给当前歌曲点亮爱心
5. 打开歌词或专辑页

### 后续高级玩法

1. 按专辑主题切换动作包
2. 按歌曲情绪切换动作速度
3. 根据播放进度触发中段彩蛋动作

## 6.4 语音交互

### 推荐定位

语音不是主入口，但应该是高级亮点能力。

### 合理的首批语音指令

1. “播放”
2. “暂停”
3. “下一首”
4. “上一首”
5. “打开音乐”
6. “播放周杰伦”
7. “播放《七里香》”
8. “播放《范特西》专辑”

### 为什么值得做

因为这个产品和普通桌宠最大的差异之一，就是它可以变成“音乐桌面助手”，而不是只会动。

## 6.5 陪伴感设计

这个产品如果只做控制，会偏工具；只做角色，会偏摆件。最好的是两者结合。

可以设计的轻陪伴方式：

1. 点击角色时，弹出一句极短的角色化反馈
2. 长时间播放同一专辑时，触发该专辑专属小动作
3. 暂停太久时，角色进入等待动作
4. 切到非周杰伦歌曲时，角色淡出或进入休眠
5. 不同专辑角色有自己的站姿和气质

重点不是话多，而是 `少量、准、好玩`。

## 7. 建议的产品结构

## 7.1 运行结构

建议做成 4 层：

1. `Music State Layer`
   负责读取 Apple Music 当前状态、曲目、专辑、播放状态

2. `Album Matching Layer`
   负责把歌曲元数据映射到专辑角色 ID

3. `Pet Runtime Layer`
   负责桌宠窗口、动作状态机、拖动、点击、动画切换

4. `Interaction Layer`
   负责点击、菜单、语音、快捷操作

## 7.2 角色资产结构

建议尽快标准化为：

1. `album_id`
2. `display_name`
3. `cover_keywords`
4. `sprite_pack`
5. `state_animations`
6. `fallback_thumbnail`

当前这 16 张图是 `角色主身份图`，后续还需要补：

1. 每个角色的状态动作图
2. 每个角色的静态缩略图
3. 每个角色的专属色板

## 7.3 专辑映射结构

建议未来维护一份映射表，字段至少包括：

1. `artist_aliases`
2. `album_aliases`
3. `album_id`
4. `pet_id`
5. `priority`
6. `notes`

这样后面即使 Apple Music 字段出现不同写法，也能稳定命中。

## 8. 体验原则

1. `先准，再炫`
   角色切换必须先准，不能专辑认错。

2. `先轻，再重`
   产品应该轻，不要像替代播放器一样压过主工作区。

3. `先自然，再复杂`
   动作少一点没关系，但必须自然、耐看、不烦人。

4. `先陪伴，再说教`
   不要做成一直打断用户的会说话玩具。

5. `先本机可控，再谈联网扩展`
   Apple Music 联动、桌宠状态、语音控制，优先保证本机闭环。

## 9. 风险与难点

## 9.1 最大技术难点

1. Apple Music 元数据与专辑名匹配可能不统一
2. 桌宠动画资源远比静态人物图多
3. 语音控制如果做得重，会把产品拖回“输入法产品”
4. 如果动作切换太频繁，桌宠会显得很烦
5. 如果直接接管播放队列，容易改变用户原本的听歌习惯

## 9.2 最大产品难点

1. 用户会不会觉得只是“好看一下”，而不是愿意长期开着
2. 如何在“可玩”与“不打扰”之间拿平衡
3. 如何让不同专辑角色真的有差异感，而不是只换皮

## 10. 推荐开发优先级

这里不做“最小方案”，而是给出一条适合做成像样产品的优先级。

### 第一阶段：跑通主体验

1. 读取 Apple Music 当前播放状态
2. 识别周杰伦专辑
3. 切换到对应角色
4. 支持拖动、点击、菜单
5. 支持播放 / 暂停 / 下一首 / 上一首

### 第二阶段：把桌宠做活

1. 给每个角色补基础动作状态
2. 加入角色切换过渡
3. 加入等待、失败、播放中等动作逻辑
4. 做 menu bar 设置入口
5. 做角色常驻与位置记忆

### 第三阶段：把它做出差异化

1. 加语音控制
2. 加专辑专属彩蛋动作
3. 加歌词 / 情绪联动
4. 加播放成功后的反馈动画
5. 加更多桌面陪伴细节

## 11. 外部项目里真正值得学的东西

### 应该学的

1. `Clawd on Desk`
   学它把“后台状态”翻译成“前台动作”的方式。

2. `SpotMenu`
   学它把 Apple Music 做成轻量伴随工具，而不是完整播放器。

3. `Tuneful`
   学它原生 macOS 音乐控制入口的产品形态。

4. `LyricFever`
   学它把歌曲信息延展成氛围层，而不是只做按钮。

5. `VPet`
   学它如何把角色、动作、内容体系做成长期资产。

### 不应该盲学的

1. 不要一开始就做成完整播放器替代品。
2. 不要一开始就做成话很多的聊天宠物。
3. 不要为了“AI 感”把本来简单的音乐控制做复杂。
4. 不要把视觉动作做得像节拍器一样持续抖动。

### 最适合你的组合

最适合你的不是单独抄某一个项目，而是组合：

1. `SpotMenu / Tuneful` 的音乐伴随层
2. `Clawd on Desk` 的状态映射思路
3. `VPet` 的角色内容体系
4. 你当前已经做完的周杰伦专辑人物资产

## 12. 代码复用建议

### 最值得直接借的部分

1. `runOsaScript(...)`
   适合作为桌宠调用 Apple Music 的基础执行层。

2. `V4MusicControlTool` 中的播放、暂停、继续、上一首、下一首逻辑
   这部分已经接近可直接抽成 `PetMusicController`。

3. `V4MusicControlTool` 中的播放验证逻辑
   这是非常值钱的部分，因为它不只是“发命令”，而是会验证“到底播没播对”。

4. `persistent ID` 精确播放
   适合后续做“点某个专辑角色 -> 播该专辑代表作”。

### 需要裁薄后再借的部分

1. 音乐意图识别
   可以保留成一个很薄的 `PetMusicIntentRouter`。

2. 权限管理
   保留麦克风、辅助功能、自动化异常判断即可。

### 暂时不要搬的部分

1. 整套输入法交互协调器
2. 大型 agent runtime
3. 非音乐域的提醒、文档、邮件、飞书能力
4. 过重的多模型配置体系

## 13. 建议的数据与模块边界

### 核心数据对象

1. `CurrentTrack`
   包含 `artist`、`album`、`title`、`persistent_id`、`playback_state`

2. `AlbumCharacter`
   包含 `album_id`、`display_name`、`aliases`、`asset_root`、`default_state`

3. `PetState`
   包含 `pet_id`、`visual_state`、`position`、`is_visible`、`last_trigger_reason`

4. `VoiceCommand`
   包含 `raw_text`、`intent`、`target_album`、`target_track`

### 建议模块

1. `MusicObserver`
   负责 Apple Music 当前状态读取和事件监听

2. `AlbumMatcher`
   负责把当前曲目映射到专辑角色

3. `PetRuntime`
   负责桌宠窗口、动作切换、拖动、点击

4. `MusicController`
   负责 Apple Music 控制与结果校验

5. `VoiceBridge`
   负责语音指令到音乐意图的转换

6. `MenuBarHub`
   负责菜单栏入口、设置、调试与快速控制

## 14. 我对这个产品的最终判断

这是一个很有潜力的方向，原因不是“桌宠”本身新，而是你手上正好有三样能形成组合优势的东西：

1. 已经完成的专辑核心人物设计
2. Apple Music 可天然触发的强内容场景
3. 本机已有可复用的音乐控制内核

所以这件事最值得做的，不是“把角色放上去会动”，而是做成：

`一个能识别周杰伦音乐语境、能表达专辑角色个性、还能轻量控制播放的桌面伴侣。`

这才是它真正像样的地方。

## 15. 本轮建议结论

如果只给一个结论，我建议后面把产品定义为：

`独立 macOS 原生桌宠 + Apple Music 本机联动 + 周杰伦专辑角色系统 + 轻语音控制`

而不是：

`Codex Pet 二次套壳`

前者有机会变成完整产品，后者更像一次有趣但边界很死的拼装。
